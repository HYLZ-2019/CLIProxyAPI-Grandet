# Version Design

用于记录用户对产品最终形态的原始想法。

---

## 更好的额度与用量记录统计分析功能

### API 秘钥机制

现在是只有一个 list 的 API 秘钥。希望给每个 API 秘钥：

1. 一个唯一的 id（1/2/3/4/....），删掉前面的 key 之后，后面 key 的唯一 id 不会变，如果新建一个 key，会往后面编号，而不是替代前面的编号。

2. 一个 string，是它的名称，可以修改和编辑，例如 "alice-work"。

---

## 分析统计与用量追踪

### 配额额度记录

- 每个小时主动去查一次每个 provider 的限额用量，存入本地数据库。
- 每次遇到 429，也把这个用尽的信号记录下来（哪个 provider、哪个 auth、哪个模型）。

### 用户使用记录

- 记录每次 query 的：模型名、input tokens、output tokens、cached input tokens、total tokens。
- 按每个 client API key 的唯一数字 ID 归属。
- 原始 per-query 数据按每小时聚合；原始日志保留 N 天后自动清理（N 可在前端配置，默认 7 天），每小时聚合数据永久保留。

### 数据库

- 使用本地 SQLite（`modernc.org/sqlite`，纯 Go，无需 CGO）。
- 数据库文件位于 `{config_dir}/analytics.db`，大小估算约 5–80 MB/月。

### 前端配置

- 在设置页面增加「Analytics」分区，包含：
  - 开关：启用/禁用分析统计。
  - 数字输入：原始日志保留天数（默认 7）。

### 前端统计展示调整

- 前端不再展示：
  - 配额封顶热力图。
  - 最近 429 事件列表。
  - 配额快照列表。
- 前端希望改为展示折线图，以及当前日期解算出的 token 价格表。

### token 价格解算范围

- 每个供应商（openai / claude / gemini / codex / antigravity 等）有不同的模型和定价。
- 不同供应商之间不混在一起计算；下面的所有计算都限定在同一个供应商内。
- 每个供应商单独用同样的方法处理，并为每个供应商画一张独立图。
- 每个模型的单价不同，而且未知。
- 同一个模型下，input / output / cached_input 三种 token 的单价也不同，而且未知。
- 价格可能随时间变化，因此需要维护「每日 token 价格」数据库。

### 通过线性方程解 token 单价

- 将 100% 周额度映射为 10000 个额度点。
- 对某个供应商，若第 X 小时查询到周用量已使用 `a%`，第 X+1 小时查询到周用量已使用 `b%`，并且这 1 小时内 CLIProxyAPI 记录到了各种模型、各种 token 类型的用量（单位为 1M token），则有一条方程：
  - 所有「模型 + token 类型」的用量，乘以各自未知单价后求和，等于 `(b - a) * 10000`。
- 收集足够多的方程后，可以解出该供应商下各模型、各 token 类型的单价。
- 如果 X 到 X+1 小时之间额度刷新，则方程不成立；解方程前筛掉所有 `b <= a` 的方程。
- 方程存在误差，因为额度可能不只通过 CLIProxyAPI 消耗，也可能通过网页端等其它渠道消耗；解方程算法需要鲁棒。

### 每日 token 价格数据库

- 每天 0 点，检查过去 24 小时内实际使用过哪些「模型 + token 类型」。
- 只针对过去 24 小时内使用过的 token 种类新增当日价格记录。
- 对这些 token 种类，往更早的历史里寻找尽可能新的有效方程，直到方程数量足以解出所有需要的值。
- 过去 24 小时没有使用过的 token 种类，不在数据库里增加当天记录。
- 前端需要展示当前日期解算出的各种 token 价格表。

### 折线图设计

- 每个供应商一张折线图。
- 横轴是时间。
- 第一条折线表示主动查询到的周额度剩余量：
  - 如果供应商返回已使用 `x%`，前端显示 `100 - x`。
  - 例如已使用 31%，则显示 69%。
  - 这样曲线预期表现为不断上升，并在每周额度刷新时归零/重置。
- 对每个 429 报错发生的时刻，在第一条折线上加一个小红点。
- 增加一个选项，可以隐藏所有 429 小红点。
- 第二条折线表示 CLIProxyAPI 当前记录到的累计 token 用量折算额度：
  - 对每种 token，查询 token 价格数据库里的最新价格记录。
  - 将 CLIProxyAPI 记录到的 token 数量按最新单价加权求和。
  - 纵轴尺度与周额度一致，最大值 10000 对应 100% 周额度。
  - 这条曲线会持续累计，并在某些时刻归零。
- 第二条折线默认归零点：每个检测到「周用量变小」的时刻。
- 增加一个选项：也可以让累计额度在每个发生 429 的时刻归零。

### 供应商统计值

- 对每个供应商，计算一个统计值：
  - 使用当前单价中最贵的 token 种类来估算。
  - 计算用尽 100% 周额度可以使用多少 M token。
